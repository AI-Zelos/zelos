# Zelos v0.8.0 需求规格说明书

> 基于 `docs/temporal-reliability-analysis.md` 的 P0-P3 优化建议，制定 v0.8.0 开发计划。
> 核心主题：**Event Sourcing（事件溯源） + 可靠性增强**——让 Zelos 的 Goal 状态从事件可重建。

---

## 一、版本目标

v0.8.0 对标 Temporal 的可靠性模型，补齐 Zelos 在以下三个维度的短板：

1. **崩溃恢复** — Goal 状态能从 Event History 完整重建（当前重启丢 Goal）
2. **重试增强** — Heartbeat Timeout + NonRetryableErrors（当前无心跳超时）
3. **事件溯源** — 事件单调递增 ID + Query 不写事件（当前 UUID + 无区分）

---

## 二、需求清单

### REQ-01：Goal 状态序列化与反序列化

**优先级：P0 | 预计工作量：2-3 天**

#### 背景
当前 `ZelosRuntime._goals` 是一个内存 `dict`，Runtime 重启后所有正在执行的 Goal 丢失。Temporal 的做法是：每个 Workflow 的状态可以从 Event History 完整重建。

#### 需求描述
1. `Task` 类增加 `to_dict()` 和 `from_dict()` 方法，完整序列化 Task 状态。
2. 新增 `GoalState` 数据类，包含 `goal_id`、`status`、`tasks`、`event_position`。
3. `GoalState` 实现 `to_dict()` / `from_dict()`。
4. `GoalState` 在每次 Goal 状态变更时通过 `StorageBackend.set_state()` 持久化一个快照。
5. Runtime 启动时自动从 Storage 恢复所有未完成的 Goal。

#### 验收标准
- [ ] `Task.to_dict()` 和 `Task.from_dict(some_dict)` 往返一致
- [ ] `GoalState.to_dict()` 和 `GoalState.from_dict(some_dict)` 往返一致
- [ ] Runtime 重启后，异步未完成的 Goal 状态恢复正确
- [ ] `tests/test_recovery.py` 覆盖：单 Task / DAG 多 Task / 重试中

---

### REQ-02：事件溯源——Goal 状态从 Event History 重建

**优先级：P0 | 预计工作量：2-3 天**

#### 背景
当前 EventBus 是"审计日志"而非"状态计算引擎"。Temporal 的做法是：Workflow State = Replay(EventHistory)。Zelos 应该能从事件链重建 Goal 状态，而不依赖内存 snapshot。

#### 需求描述
1. 在 `EventBus` 或新增的 `EventSourcingEngine` 中实现 `apply_event(event, state) -> state` 纯函数。
2. 为 `goal.submitted`、`plan.created`、`task.created`、`task.completed`、`task.failed`、`goal.completed` 等核心事件类型编写 handler。
3. `restore_goal(goal_id, storage) -> GoalState` — 读取事件链 + 前进到最新 snapshot 位置 + 重放剩余事件。
4. 快照 + 增量事件的组合恢复：不从第 0 个事件开始重放。

#### 验收标准
- [ ] 从空状态 + 事件链重建 GoalState，与原始状态一致
- [ ] 快照 + 增量事件恢复，重建状态与原始一致
- [ ] 100 个事件的重放 < 100ms
- [ ] `tests/test_event_sourcing.py` 覆盖：完整重建、快照+增量、空事件链

---

### REQ-03：事件单调递增 sequence_id

**优先级：P2 | 预计工作量：0.5 天**

#### 需求描述
1. `Event` 增加 `sequence_id: int` 字段，单调递增。
2. `InMemoryEventStore` 首次插入时自动分配 `sequence_id`。
3. `replay_from(sequence_id)` 按 `sequence_id` 定位。

#### 验收标准
- [ ] 连续发布 100 个事件，`sequence_id` 分别为 0-99
- [ ] `replay_from(50)` 返回后 50 个事件
- [ ] 新旧事件共存时 `sequence_id` 向后兼容（缺省为 -1）

---

### REQ-04：Heartbeat Timeout 检测

**优先级：P1 | 预计工作量：1 天**

#### 需求描述
1. `InFlightTask` 增加 `heartbeat_at` 和 `heartbeat_timeout_ms` 字段。
2. `ExecutionEngine` 的 monitor 线程增加心跳超时检测逻辑。
3. 心跳超时 → Task 标记为 FAILED → Scheduler 评估重试。
4. Agent 在执行过程中可以通过 `submit_heartbeat(task_id)` 发送心跳。

#### 验收标准
- [ ] Agent 在 30s 内未发送心跳 → Task 标记 FAILED → 自动重试
- [ ] Agent 定期发送心跳 → Task 正常完成
- [ ] `tests/test_heartbeat.py` 覆盖：正常心跳、心跳超时、重试后成功

---

### REQ-05：NonRetryableError 支持

**优先级：P1 | 预计工作量：0.5 天**

#### 需求描述
1. `Task` 增加 `non_retryable_errors: list[str]` 字段。
2. Agent 返回 error 时，如果 `error.code` 在 `non_retryable_errors` 中 → 直接标记 `FAILED`（不可重试）。
3. `TaskStatus` 增加 `FATAL_FAILED` 状态（不可重试的失败）。

#### 验收标准
- [ ] `non_retryable_errors = ["ValidationError"]` → Agent 返回 ValidationError → Task FATAL_FAILED
- [ ] 不在列表中的错误 → 正常重试

---

### REQ-06：Query 操作不写入事件

**优先级：P2 | 预计工作量：0.5 天**

#### 需求描述
1. `get_goal_status()`、`list_agents()`、`get_health()` 等查询操作不产生 EventBus 事件。
2. 当前这些操作已不写事件——本需求是确认规范。

#### 验收标准
- [ ] 调用 `get_goal_status()` 后 EventBus 事件数不增加
- [ ] 规范文档更新

---

### REQ-07：Retry 历史追踪

**优先级：P2 | 预计工作量：0.5 天**

#### 需求描述
1. 每次重试发布 `task.retry_scheduled` 事件到 EventBus。
2. 事件 payload 包含：`task_id`、`attempt`、`backoff_ms`、`previous_error`。
3. 在 `get_goal_status()` 中可查询每个 Task 的重试历史。

#### 验收标准
- [ ] Task 重试时 EventBus 收到 `task.retry_scheduled` 事件
- [ ] 事件 payload 包含完整重试上下文

---

## 三、不纳入 v0.8.0 的内容

以下内容来自 Temporal 分析但**不在 v0.8.0 范围**：

| 项目 | 原因 |
|------|------|
| SideEffect 接口隔离 + TestMode | Temporal 的确定性回放机制不适用于非确定性 Agent |
| 确定性执行约束 | Zelos 的设计哲学是接受 Agent 非确定性 |
| 不同 Task 不同重试策略 | zelos.yaml 配置复杂度收益不高 |
